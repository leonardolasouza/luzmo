# Security Standards: Luzmo Embedded Dashboard PoC

## 1. Overview
This document outlines the security posture and requirements for the Luzmo Embedded Dashboard Proof of Concept (PoC). Given the nature of embedded analytics, the primary security focus is strictly preventing unauthorized access to the Luzmo account by securing the core API credentials.

## 2. Core Security Principles

### 2.1 Zero Client-Side Secrets
Under no circumstances should the Luzmo `API_KEY` or `API_TOKEN` be sent to, stored in, or processed by the client browser. 
- The frontend architecture follows a "Dummy Client" model.
- The frontend is only aware of the temporary, short-lived authorization tokens provided by the backend.

### 2.2 Server-Side Token Generation
All authentication with the Luzmo platform occurs on the backend (Node.js/Express). 
- The backend acts as a secure intermediary.
- It uses the official `@luzmo/nodejs-sdk` to securely communicate with Luzmo's authentication servers.
- The backend requests an SSO (Single Sign-On) authorization token and forwards only this temporary token to the frontend.

## 3. Threat Mitigation

### 3.1 Credential Leakage via Network
- **Mitigation:** The frontend relies on an internal `/api/v1/token` endpoint (pointing to [Constants.md](./Constants.md)). The backend processes the request and responds strictly with the temporary `{ id, token }` authorization object.
- **Validation:** Automated backend integration tests (using Jest/Supertest) and/or E2E tests (using Playwright) must be run to explicitly assert that environment secrets are never serialized into HTTP responses. See the [Testing PRD](./Testing.md) for details.

### 3.2 Credential Leakage via Source Control
- **Mitigation:** For this local dev/PoC environment, all secrets must be stored locally in a `.env` file. *Note: In a production environment, hardcoded `.env` files should be replaced with a secure Cloud Secret Manager (e.g., AWS Secrets Manager, Google Secret Manager, or Azure Key Vault) to inject secrets directly into the runtime.*
- **Validation:** The `.env` file must be explicitly declared in the `.gitignore` file to ensure API keys are never pushed to the Git repository.

### 3.3 Stolen Client Tokens
- **Mitigation:** The SSO tokens generated by the backend are strictly temporary.
- **Validation:** The backend SDK call explicitly uses the `inactivity_interval` parameter (e.g., `3600` seconds). If a malicious actor intercepts the client token, its usability window is severely limited.

## 4. Future Enhancements (Multi-Tenancy & Row-Level Security)
While the current PoC is a simple implementation embedding a single dashboard, Luzmo's architecture supports robust multi-tenancy.
- **Future Implementation:** To ensure different users only see the data they are allowed to see, the backend will validate the requesting user's identity against the primary application's database.
- **Luzmo Integration:** The backend will pass specific metadata (like a `Tenant ID` or user-specific filters) to the Luzmo SDK when requesting the SSO token. 
- **Result:** Luzmo generates an authorization token cryptographically bound to those filters, enforcing row-level security (RLS) entirely at the Luzmo server level. The client cannot manipulate the token to view unauthorized data.
